When an inference server feels slow, the first instinct is almost always to raise the batch size. Sometimes that helps. Often it does nothing, or makes things worse — because the server was never compute-bound to begin with. It was memory-bound, and the memory in question is the KV cache.

What the KV cache actually costs

Every token generated adds a new key and value vector, per layer, per attention head, that has to stay resident for the lifetime of the sequence. For a mid-size model with a few dozen layers and a handful of billion parameters, that's easily tens of kilobytes of cache per token. Multiply by a few thousand tokens of context, times however many sequences you're serving concurrently, and the cache footprint dwarfs the model weights themselves.

This is the part that doesn't show up in a naive throughput benchmark run at batch size one. It only shows up once you're serving real traffic with long, variable-length contexts — which is exactly when people start asking why GPU utilization looks fine but latency is climbing anyway.

Fragmentation, not just size

The more subtle failure mode isn't raw cache size, it's fragmentation. Naive implementations allocate one contiguous block per sequence, sized for the maximum possible length. Short sequences waste the unused tail; finished sequences leave holes that don't get reused cleanly. The result is a GPU that reports plenty of free memory in aggregate while simultaneously rejecting new requests because no single contiguous block is large enough.

This is the exact problem paged attention was built to solve: treat the KV cache like virtual memory, allocate it in fixed-size blocks, and let a page table stitch blocks together per sequence instead of requiring contiguity. It trades a small amount of indirection overhead for the ability to pack memory far more tightly — which in practice is a very good trade.

How to actually tell which one you have

Before reaching for a bigger batch size or a fancier scheduler, it's worth separating the two failure modes:

  • Watch GPU memory and request queue length together. If memory is near-saturated while requests queue up, you're cache-bound, not compute-bound.
  • Compare throughput at fixed batch size across short vs. long average context lengths. If throughput collapses as context grows even though compute per token barely changed, that's cache pressure, not FLOPs.
  • Log allocation failures, not just OOMs. A server that's silently falling back to smaller batches because it can't find contiguous space is a fragmentation problem wearing a throughput problem's clothes.

The practical takeaway

Batch size is a lever on compute. Once your workload has any meaningful spread in context length, memory layout becomes the lever that actually matters. Reach for paging and better cache eviction policies before reaching for more GPUs — it's usually the cheaper fix, and it's the one that actually addresses what's throttling you.
